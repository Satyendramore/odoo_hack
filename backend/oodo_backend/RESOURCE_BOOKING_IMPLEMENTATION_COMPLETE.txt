CORE FEATURES IMPLEMENTED
================================================================================

1. HALF-OPEN INTERVAL OVERLAP DETECTION
   Status: ✓ Implemented and Verified
   
   Query Logic:
     startTime < endTime2 AND endTime > startTime2
   
   Examples:
     ✓ 9:00-10:00 + 10:00-11:00 = NO OVERLAP (back-to-back allowed)
     ✓ 9:00-10:00 + 9:30-10:30 = OVERLAP (rejected)
     ✓ 9:30-10:30 + 9:00-10:00 = OVERLAP (rejected)
   
   Why Important:
     - Users often book consecutive time slots (e.g., meetings back-to-back)
     - Must allow 0-second gaps between bookings
     - Using <= or >= would incorrectly reject valid bookings

2. PESSIMISTIC WRITE LOCKING
   Status: ✓ Implemented
   
   Implementation:
     @Lock(LockModeType.PESSIMISTIC_WRITE)
     @Query("SELECT b FROM Booking b WHERE ...")
     List<Booking> findOverlapping(...);
   
   Concurrency Protection:
     - Thread A checks overlap with lock held
     - Thread B waits at database level
     - Thread A creates booking, releases lock
     - Thread B checks again, finds conflict, throws 409
   
   Result: Only one booking per time slot possible

3. SERIALIZABLE TRANSACTION ISOLATION
   Status: ✓ Implemented
   
   Applied To:
     - book() method
     - reschedule() method
   
   Guarantees:
     - Transactions appear to execute serially
     - No dirty reads, non-repeatable reads, or phantom reads
     - Combined with pessimistic lock: maximum safety

4. ATOMIC RESCHEDULE OPERATION
   Status: ✓ Implemented
   
   Implementation:
     @Transactional(isolation = SERIALIZABLE)
     public reschedule(id, newRequest, user) {
         cancel(id, user);           // Within transaction
         return book(newRequest, user);  // Within transaction
     }
   
   Guarantee:
     - Both operations succeed or entire transaction rolls back
     - No partial state (e.g., old cancelled but new not created)

5. AUTHORIZATION AT SERVICE LAYER
   Status: ✓ Implemented
   
   Who Can Cancel:
     ✓ Person who made the booking
     ✓ Their Department Head
     ✓ Asset Manager
     ✓ Admin
   
   Enforcement:
     - Service layer checks before any operation
     - Not just controller-level @PreAuthorize
     - Defense in depth principle

6. ASSET BOOKABILITY VALIDATION
   Status: ✓ Implemented
   
   Check:
     if (!asset.isBookable()) {
         throw AssetNotBookableException;
     }
   
   Use Case:
     - Some assets may not support time-slot booking
     - E.g., fixed installation equipment
     - Returns 400 Bad Request if not bookable

7. STRUCTURED ERROR RESPONSES
   Status: ✓ Implemented
   
   409 Conflict Response:
     {
       "timestamp": "2025-07-12T14:30:00",
       "status": 409,
       "message": "Requested slot ... overlaps with existing booking ...",
       "conflictingStart": "2025-07-15T14:00:00Z",

     }
   
   Frontend Use:

CONCURRENCY SAFETY GUARANTEE
================================================================================

Scenario: Two concurrent booking requests for overlapping time slots

Setup:
  Asset: AF-0042 (Conference Room)
  User A: Requests book 14:00-15:00
  User B: Requests book 14:30-15:30 (concurrent, overlaps)

Execution (Pessimistic Lock):
  T0: User A → POST /bookings
      ├─ Hits findOverlapping() with @Lock(PESSIMISTIC_WRITE)
      ├─ Database acquires exclusive lock on booking rows for AF-0042
      ├─ Check finds no conflicts (first booking)
      └─ Creates booking, transaction commits → lock released

  T0: User B → POST /bookings (concurrent with T0)
      ├─ Queued at database level (waiting for lock)
      └─ (blocked until User A's lock released)

  T1: User A's transaction completes, lock released
      └─ User B's query proceeds

  T1: User B → Resumes findOverlapping() query
      ├─ Re-acquires lock
      ├─ Finds User A's booking
      ├─ Checks overlap: 14:30 < 15:00 AND 15:30 > 14:00 → TRUE
      └─ Throws BookingOverlapException

  T2: User B receives 409 Conflict response
      └─ conflictingStart: 14:00, conflictingEnd: 15:00

Result:
  ✓ User A: 201 Created (booking successful)
  ✓ User B: 409 Conflict (overlap detected)
  ✓ No duplicate bookings created
  ✓ Race condition prevented

What Makes This Safe:
  1. Pessimistic Lock: Database enforces serialization
  2. SERIALIZABLE Isolation: Transactions don't interfere
  3. Half-Open Interval: Correct overlap detection
  4. Atomic: All-or-nothing operation

1. BookingOverlapException (HTTP 409 Conflict)
   Thrown: When booking request overlaps with existing booking
   Includes: conflictingStart and conflictingEnd for UI highlighting
   Response: BookingOverlapErrorResponse with conflict details
   
2. AssetNotBookableException (HTTP 400 Bad Request)
   Thrown: When trying to book asset with isBookable = false
   Example: "Asset AF-0042 is not available for booking"
   Response: Standard ErrorResponse
   
3. BookingNotFoundException (HTTP 404 Not Found)
   Thrown: When booking ID doesn't exist
   Example: "Booking with ID ... not found"
   Response: Standard ErrorResponse
   
4. InvalidBookingStateException (HTTP 400 Bad Request)
   Thrown: When trying to cancel already-cancelled/completed booking
   Example: "Cannot cancel booking with status COMPLETED"
   Response: Standard ErrorResponse

All handlers added to GlobalExceptionHandler.java
