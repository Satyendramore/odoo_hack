KEY FEATURES IMPLEMENTED
=============================================================================

1. HALF-OPEN INTERVAL OVERLAP DETECTION
   - Logic: startTime < endTime2 AND endTime > startTime2
   - Back-to-back bookings allowed (gap = 0 seconds)
   - Overlapping bookings rejected with 409 Conflict

2. PESSIMISTIC WRITE LOCKING
   - Query uses @Lock(LockModeType.PESSIMISTIC_WRITE)
   - Database applies SQL FOR UPDATE clause
   - Prevents race conditions during concurrent booking attempts
   - Lock held until transaction commits

3. SERIALIZABLE TRANSACTION ISOLATION
   - Transactions appear to execute serially
   - Combined with pessimistic lock for maximum safety
   - Applied to book() and reschedule() methods

4. ATOMIC RESCHEDULE OPERATION
   - Cancel old booking + Create new booking in single transaction
   - Either both succeed or entire transaction rolls back
   - No partial states (old cancelled + new not created scenario)

5. AUTHORIZATION AT SERVICE LAYER
   - Only bookedBy, their dept head, or asset manager/admin can cancel
   - Verified inside service method (not just controller)
   - Defense in depth: controller + service layer checks

6. STRUCTURED ERROR RESPONSES
   - 409 Conflict includes conflictingStart and conflictingEnd times
   - Frontend can highlight conflicting slots on calendar UI
   - Helpful, actionable error messages

7. ASSET BOOKABILITY VALIDATION
   - Checks asset.isBookable = true before allowing booking
   - 400 Bad Request if asset not bookable
   - Some assets may not support time-slot booking

8. DTO VALIDATION
   - BookingRequest: @NotNull, @Future validations
   - Class-level check: endTime must be strictly after startTime
   - Validation errors caught before service layer

CONCURRENCY SAFETY VERIFICATION
=============================================================================

Test Scenario: Two concurrent requests for overlapping slots
  Asset: AF-0042 (Conference Room)
  Alice: Book 14:00-15:00
  Bob: Book 14:30-15:30 (concurrent, overlaps)

Expected Behavior:
  - Request 1 (Alice): Succeeds with 201 Created
  - Request 2 (Bob): Fails with 409 Conflict
    Response includes Alice's booking times for UI

Why It Works:
  1. Alice's book() call hits findOverlapping() with PESSIMISTIC_WRITE lock
  2. Database locks the booking rows for AF-0042
  3. Alice's check finds no conflicts (first booking)
  4. Alice creates booking, transaction commits
  5. Bob's concurrent book() call queues at database level
  6. Lock released after Alice's transaction commits
  7. Bob's query resumes, finds Alice's booking
  8. findOverlapping() returns [Alice's booking]
  9. Bob's service throws BookingOverlapException
  10. 409 Conflict response sent to Bob

Edge Case: Back-to-Back Bookings (Allowed)
  Alice: Book 14:00-15:00
  Charlie: Book 15:00-16:00 (starts when Alice ends)
  
  Query: 15:00 < 16:00 AND 15:00 > 15:00 → false (NO OVERLAP) ✓
  Result: Both bookings succeed (back-to-back is OK)
